# 풀탭 (당기면 움직이는 장치) (pull-tab)

## 개요

카드 앞면에 뚫린 좁은 슬롯(트랙) 안에서 별도로 오린 슬라이더 조각을 손잡이로 밀거나 당기면 그림이 옆으로 이동하는 메커니즘. `sliderWidth`/`sliderHeight`는 슬라이더 조각의 크기(mm), `trackLength`는 카드에 뚫는 슬롯의 길이(mm)다. 기본값은 `sliderWidth=30, sliderHeight=15, trackLength=80`.

이 메커니즘은 다른 슬라이더형 메커니즘(rising-slide, auto-slide-window, slide-to-swing 등)과 달리 "카드 열림 각도"로 구동되는 3D 조립 포즈 프리뷰가 없다 — 애초에 슬라이더가 수평 이동만 하는 자유도라 각도 기반 포즈 모델과 궁합이 다르다. 또한 다른 여러 메커니즘과 달리 **실제 자르기/접기 기하 버그를 고친 전용 수정 기록이 없다**(아래 "이전 작업" 절 참고) — 이 메커니즘은 실행 가능해진 이후 실제 기하 검증을 거친 기록이 없는 상태다.

## 작동 방식

핵심 공식: `travel = trackLength - sliderLength - 2·buffer`, `slotWidth = paperThickness + clearance (0.5~1.0mm)`.

파라미터 검증(clamp): `sliderWidth` 8~60mm, `sliderHeight` 5~40mm, `trackLength` 20mm~(카드 폭 - 인쇄 안전 여백×2 - 20), `paperThickness` 기본 0.3mm, `clearance` 0.5~1.5mm(기본 0.8), `buffer` 1~5mm(기본 2).

4개의 독립된 조각을 계산한다:

1. **트랙 슬롯(track slot)** — 카드에 뚫는 좁은 직사각형 구멍. `slotWidth = round(paperThickness + clearance, 1)`, `sliderLength = sliderWidth`(슬라이더는 자신의 폭 방향으로 이동), `travel = round(trackLength - sliderLength - 2·buffer, 1)`. 트랙 중심은 기본적으로 카드 상단 절반(척추 위쪽)에 위치. 네 변을 계산해 CUT으로 그린다.
2. **안내 띠(guide strips)** — 슬롯 위아래에 카드 뒷면에 붙이는 폭 4mm짜리 별도 조각 두 개. 트랙보다 살짝 길게(`trackLength + 6`) 잘라 슬롯 양옆을 감싸며, 가운데에 접는 선 하나씩을 그어 "슬라이더를 덮듯이 접어 넣는" 형태로 설계돼 있다. 이 띠는 슬라이더를 정렬 상태로 유지하기 위해 카드 뒷면에 접착하는 용도로 추정된다(소스 자체에 명시적 확언은 없다).
3. **슬라이더 조각(slider piece)** — 카드 인쇄 영역 아래 여백에 배치되는 별도 몸체. 몸체(사각형)에 손잡이 탭(폭 8mm, 사다리꼴로 바깥쪽이 넓어지는 모양)이 이어 붙고, 몸체-손잡이 경계에 골접기(VALLEY_FOLD) 선 하나.
4. **멈춤 탭(stops)** — 슬라이더 좌우 끝 위아래에 작은 돌기 4개(폭 3mm, 높이 1.5mm) — 안내 띠에 걸려 슬라이더가 슬롯 밖으로 완전히 빠지지 않게 하는 역할로 추정된다(주석에 명시적 서술은 없음).

각 조각의 위치/치수를 설명하는 마커 텍스트(슬롯 크기, 이동 거리, 슬라이더 크기, 손잡이)도 함께 생성한다. 최종 렌더링은 트랙 슬롯/안내 띠/슬라이더/멈춤 탭 네 개의 별도 그룹으로 나눠 각각의 스타일(CUT/MOUNTAIN_FOLD/VALLEY_FOLD)로 그린다 — 이 네 그룹 분리는 이 조각들이 **물리적으로 서로 다른 잘라낼 부분**(카드 자체의 트랙 구멍 vs. 별도로 오려서 조립하는 슬라이더/가이드 조각들)이라는 것을 반영한다.

## 이 프로젝트에서의 활용

3D 조립 포즈 프리뷰가 연결되어 있지 않다 — 항상 "3D 미리보기 준비 중" 플레이스홀더가 표시된다. 인쇄용 조립 설명서도 전용 문구 없이 범용 3단계("오리기 → 접기 → 풀칠") 설명뿐이다 — 슬롯/슬라이더/안내띠/멈춤탭처럼 부품이 4종류나 되는 비교적 복잡한 메커니즘임에도, 안내 띠를 "양 끝만 붙이고 가운데는 붙이지 않는다"는 식의 다른 메커니즘(rising-slide, auto-slide-window, slide-to-swing)에 존재하는 안전장치 관련 경고 문구가 이 메커니즘에는 전혀 없다.

## 이전 작업에서 배운 교훈

pull-tab의 히스토리는 다른 유사 메커니즘들과 뚜렷하게 다르다.

- **자리표시자 단계**: 한 커밋에서 화면 미리보기의 pull-tab 지원은 실제로는 몇 줄짜리 **자리표시자**였다. 코드 주석 자체가 이를 인정했다: "실제 생성기가 아직 연결되지 않았으므로 기본 pull tab 자리표시자". 즉 이 시점에는 트랙/슬라이더/안내띠/멈춤탭을 계산하는 진짜 로직이 UI에 연결돼 있지 않았다.
- 그런데 실제 생성 로직 자체는 **프로젝트 초기 커밋에서 이미 완성된 모듈로 존재**했고, 그 이후 지금까지 **단 한 번도 다시 수정된 적이 없다**(오직 최초 추가뿐). 즉 이 파일의 실제 기하 계산 로직은 프로젝트 시작 시점에 통째로 작성된 뒤 검증도, 버그 수정도 거친 적이 없다.
- **도달 가능하게 만든 커밋**: 이미 사용하고 있던 인쇄 템플릿 헬퍼와 선 스타일 헬퍼가 그 시점까지 코드베이스에 존재하지 않아, 이 생성기를 호출하면 즉시 런타임 에러가 났다 — 그래서 화면은 대신 조용히 자리표시자만 그리고 있었다. 이후 이 누락된 헬퍼들을 추가하고 메커니즘 레지스트리로 배선을 새로 해서 실제 렌더러가 처음으로 실행되게 만들었다.
- **이후 pull-tab 전용 기하(cut/fold) 버그 수정 기록이 없다.** box-popup과 parallel-fold는 실행 가능해진 뒤 실제 자르기/접기 로직에 있던 버그(CUT과 FOLD가 같은 좌표에 중복 등록되는 버그)가 발견되어 고쳐졌지만, pull-tab은 실행 가능해진 이후 지금까지 **누구도 실제 기하 검증을 거친 기록이 없다.** 코드 안에도 이 사실을 뒷받침하는 정황이 있다: 안내 띠와 멈춤 탭의 역할을 설명하는 주석이 "~하는 용도로 추정" 수준으로 약하게 서술돼 있고, 트랙 슬롯의 CUT 사각형이 카드 자체에 뚫리는 구멍임에도 이 구멍이 카드의 트림/척추 라인과 겹치지 않는지, 안내 띠가 실제로 슬라이더의 멈춤 탭에 걸리는 위치에 오는지(트랙 슬롯과 슬라이더 몸체가 서로 다른 좌표계를 쓴다) 등이 실측/실물 검증된 적이 없다.

## 앞으로 작업 시 주의사항

- **pull-tab을 수정하거나 파라미터 범위를 넓히기 전에, box-popup/parallel-fold가 겪었던 "CUT과 FOLD가 동일 좌표에 중복 등록"되는 버그 패턴이 여기에도 있는지 처음으로 점검한다.** 이 메커니즘은 그 검증을 아직 한 번도 받은 적이 없다.
- **트랙 슬롯(카드에 뚫는 구멍)과 슬라이더/안내띠/멈춤탭(별도로 오리는 조각들)의 좌표계가 서로 다르다는 점에 주의한다.** 트랙은 카드 중심 기준 좌표계, 슬라이더는 카드 아래 여백 기준 좌표계를 쓴다 — 새 기능(예: 안내 띠가 실제로 멈춤 탭 위치와 정렬되는지 확인하는 로직)을 추가할 때 두 좌표계를 혼동하지 않는다.
- **3D 조립 포즈를 추가하는 작업을 하게 된다면**, v-fold/box-popup/parallel-fold와 달리 pull-tab은 "카드 열림 각도 α"로 표현되는 접기 포즈가 아니라 "슬라이더의 수평 이동 거리(0~travel)"라는 **전혀 다른 자유도**를 가진 메커니즘이라는 점을 먼저 설계해야 한다 — 기존 회전 기반 접기 공식을 그대로 재사용할 수 없고, 수평 이동(translate) 계열의 새로운 포즈 파라미터화가 필요하다.
- **안내 띠가 "양 끝만 붙이고 가운데는 띄운다"는 식의 안전장치 설계 의도가 있는지 코드만으로는 확실치 않다** — 다른 슬라이더형 메커니즘은 조립 설명에 "가운데는 붙이지 마세요, 양 끝만 붙이세요" 같은 명시적 경고가 있는데, pull-tab은 범용 문구뿐이라 이런 조립 안전 수칙이 사용자에게 전혀 전달되지 않는다. 실물로 조립해 보고 안내 띠 접착 방식이 실제로 중요한 제약이라면, pull-tab 전용 조립 안내를 새로 만들어 PDF용 텍스트와 화면용 일러스트 양쪽에 추가해야 한다(둘 다 손으로 동기화해야 하는 구조이므로 하나만 고치면 드리프트가 생긴다).
- **파라미터 검증(clamp) 범위가 서로 상호작용하는지 실측 확인이 필요하다**: 예를 들어 `sliderWidth`를 최대값(60mm)으로, `trackLength`를 최소값(20mm) 근처로 설정하면 `travel = trackLength - sliderLength - 2·buffer`가 음수가 될 수 있는데, 이 값이 음수일 때 트랙/슬라이더 기하가 어떻게 깨지는지(또는 안전하게 처리되는지) 코드상 명시적 가드가 없다.
