# 브이폴드 (v-fold)

## 개요

카드 척추(spine) 한가운데서 V자 모양으로 갈라진 두 팔이 카드를 열면 산 모양으로 솟아오르는 가장 기본적인 팝업 메커니즘이다. 가장 단순한 팝업 위상(topology)으로, 척추 위 한 점에서 좌우로 뻗은 두 팔이 각자의 카드 면에 붙고, 정점(apex)에서 만난다. `armLength`는 V자 팔의 길이(mm), `angle`은 척추 기준 팔의 반각(half-angle, 도 단위)이며, 기본값은 `armLength=40mm, angle=45°`다.

## 작동 방식

### 2D 평면 도안

2D 도안 생성기는 항상 고정된 `angle`(기본 45°) 하나로 평면 전개도(자르고 접기 전 상태)만 그린다 — 카드가 열리는 정도에 따라 팔의 벌어짐이 달라지는 동적 렌더링은 하지 않는다.

팔 끝점은 다음과 같이 계산한다: 척추 중심 `(cx, cy)`에서 반각 `angleRad`만큼 벌어진 두 방향으로 `armLength`만큼 뻗은 점이 V자의 양 끝이다.

```
angleRad = angle(deg) → rad
dx = armLength · cos(angleRad)
dy = armLength · sin(angleRad)
leftX = cx - dx,  rightX = cx + dx,  topY = cy - dy
```

선 구성:
- **자르기(CUT)**: 중심 `(cx,cy)`에서 `(leftX,topY)`, `(rightX,topY)`로 이어지는 두 직선 — V자 팔 자체의 실루엣.
- **척추 골접기(VALLEY_FOLD)**: `(cx,topY)`~`(cx,cy)` — 척추를 따라 팝업이 접히는 선.
- **산접기(MOUNTAIN_FOLD)**: `baseFoldY = topY - 15`로 임의로 15mm 띄운 뒤, 각 팔 끝에서 `(cx, baseFoldY)`로 이어지는 두 선. 이 15mm 오프셋과 그 각도의 기하학적 근거는 소스에 검증되어 있지 않다 — 실제로 이 참고 구현에는 "표준 V-fold를 중심에서 자르는 방법은..."이라며 확신 없이 실토하는 주석이 남아 있었다. 수정 시 실제로 종이가 이 선을 따라 접었을 때 팔이 자연스럽게 세워지는지 실물로 확인이 필요하다.
- **풀칠 탭(GLUE_TAB)**: 팔 좌우에 사다리꼴로 하나씩, `'풀칠'` 라벨.

### 3D 조립 포즈 (열림 각도에 따른 동적 포즈)

2D 평면 도안과는 별개로, "카드를 α도 열었을 때" 팔이 어떤 자세를 취하는지 계산하는 동적 포즈 모델이 존재한다. 이 모델은 2D 생성기의 좌표를 재사용하지 않고, 별도의 공식으로 각도만 재해석한다. 공유하는 파라미터는 `armLength` 하나뿐이다.

좌표계: x=오른쪽, y=아래, z=시청자 쪽. 척추는 x=0 수직선, 카드 평면은 z=0.

- `α` = 카드 열림 각도(0~180°), `a = α/2`.
- `θ_page = (180-α)/2 = 90-a` — 각 페이지가 척추를 축으로 `rotateY(±θ_page)` 회전.
- `β = α`(대칭), `h = L·sin(a)`(팝업 높이), `γ = arcsin(h/L) = a`.
- **핵심 설계 포인트**: 정점(apex)은 대칭평면 위 월드좌표 `A = (0, -L·cos a, L·sin a)`에 있어야 하고, 좌우 두 팔의 끝이 항상 이 A에서 만나야 한다. 페이지의 `rotateY(±θ)`와 팔의 자체 접힘은 **서로 다른 축**에 대한 회전이라 각도를 단순히 더할 수 없다. 그래서 각 팔을 자기 페이지의 실제 DOM 자식으로 두어 부모의 `rotateY`를 상속받게 하고, 그 위에 자기만의 회전을 얹는 접근을 쓴다:
  ```
  left  arm:  rotate3d(-sin a, 0, -cos a, a°)
  right arm:  rotate3d(-sin a, 0, +cos a, a°)
  ```
  페이지의 회전 피벗과 팔의 회전 피벗이 같은 척추 중심점 O이기 때문에 어느 α에서든 두 팔 끝이 정확히 같은 A에 도달한다. α=0일 때 팔은 자기 페이지와 함께 x=0 평면 속으로 들어가 완전히 숨는다(edge-on으로 보여 사실상 보이지 않음).
  - **의도적 타협**: 단일 고정 크리스(crease) 위의 이상적인 강체 거싯(gusset)은 `h=L·sin a`를 정확히 만족시키면서 동시에 정점을 대칭평면에 유지할 수 없으므로, 매 프레임마다 접힘 축을 다시 계산해서 h와 "끝점이 만난다"는 조건만 정확히 지킨다 — 팔의 자유로운 바깥쪽 모서리는 미세하게 미끄러지지만 이 단순화된 포즈 프리뷰에서는 육안으로 티가 나지 않는다.

## 이 프로젝트에서의 활용

mypopworld에서는 이 3D 조립 포즈 프리뷰가 지원되는 3종(v-fold, box-popup, parallel-fold) 중 하나이고, 나머지 10종은 아직 3D 프리뷰가 없다. 인쇄용 조립 설명서는 전용 문구 없이 "오리기 → 접기(산/골) → 풀칠하여 조립하기"라는 범용 3단계 설명을 그대로 쓴다(v-fold 전용 조립 팁/삽화는 없음).

## 이전 작업에서 배운 교훈

3D 프리뷰를 처음 추가했을 때, **카드를 닫아도(α=0) 팝업 팔이 계속 화면에 보이는 버그**가 있었다.

근본 원인: 최초 구현에서 팔(arm) 요소들이 페이지(page) 요소의 **형제(sibling)** 노드였다 — 좌우 팔의 끝이 공유 정점에서 만나도록 하기 위해 일부러 페이지에서 분리해 별도 컨테이너 아래 두었다. 그런데 `rotateY(gamma)/rotateY(-gamma)`라는 단일 축 회전만 적용했기 때문에, 페이지가 닫혀도 팔은 페이지의 회전을 전혀 상속받지 못하고 월드 좌표계의 정지 자세를 유지했다 — 즉 α=0에서 페이지는 정확히 접히는데 팔은 그대로 카메라를 향한 채 남아 있었다.

수정: 각 팔을 실제로 자기 페이지의 DOM 자식으로 재배치해 페이지의 회전을 상속받게 하고, 그 위에 얹을 팔 자신의 회전을 `rotate3d(-sin a, 0, ∓cos a, a°)`로 다시 유도했다(정점이 여전히 같은 월드 좌표 A에서 만나도록 축을 매 프레임 재계산). 수정 후 α = 0/30/60/90/120/150/180 전 구간에서 팁 간격(tip gap)이 0.00mm임을 수치로도, 스크린샷으로도 검증했다.

## 앞으로 작업 시 주의사항

- **닫힌 상태(α=0)에서 3D 프리뷰가 실제로 안 보이는지 반드시 확인한다.** 슬라이더를 0으로 내렸을 때 팔이 edge-on으로 사라지는지 직접 확인 — 위 버그가 정확히 이 케이스였다.
- **팔/플랩을 자기 페이지의 DOM 자식으로 둘 것.** 페이지와 형제 관계로 분리하면 회전 상속이 끊기고 위와 동일한 버그가 재발한다. 이 부모-자식 nesting 규칙은 다른 유사 위상 메커니즘(box-popup/parallel-fold)에도 동일하게 적용된다.
- **두 팔의 회전 축이 같은 피벗(척추 중심 O)을 공유하는지 검증한다.** 각도만 맞고 피벗이 어긋나면 좌우 팔 끝이 벌어지는(정점이 갈라지는) 시각적 버그가 생긴다. 여러 α 값(0/45/90/135/180)에서 팁이 실제로 만나는지 확인.
- **2D 도안의 산접기 위치(`baseFoldY = topY - 15`)는 임의 상수로, 근거가 확실치 않다.** `armLength`/`angle`을 크게 바꿔 사용하는 기능을 추가하기 전에, 이 15mm 오프셋이 실제 종이 접기에서 유효한 범위인지(팔이 접혔을 때 척추 위 다른 요소와 겹치지 않는지, 각도가 지나치게 좁거나 넓을 때도 유효한지) 실물 프린트로 검증해야 한다.
- **2D 생성기와 3D 프리뷰는 서로 다른 소스에서 각도를 계산한다** — 2D는 고정된 `angle` 파라미터로 정적 전개도만 그리고, 3D는 열림 각도에 따른 동적 포즈를 그린다. 두 값의 "의미"가 다르다는 것(2D의 `angle`=팔의 벌어진 반각, 3D의 `α`=카드 열림 각도)을 혼동하지 않도록 주의.
- **v-fold 전용 조립 설명 문구/삽화가 없다** — 범용 문구를 모든 유사 메커니즘이 공유하므로, v-fold만의 조립 팁(예: 팔 각도가 좁을 때 풀칠 요령)을 추가하려면 새 조립 안내 항목을 만들고, PDF용 텍스트와 화면용 일러스트 양쪽을 함께 고쳐야 한다(둘은 손으로 동기화되는 구조라 하나만 고치면 어긋난다).
