# 상자 팝업 (box-popup)

## 개요

카드를 열면 척추 양옆에서 직사각형 상자 면이 수직으로 솟아올라 하나의 이어진 상자 앞면을 이루는 메커니즘. `width`는 상자의 폭(척추 방향, mm), `height`는 상자의 높이이자 척추로부터의 깊이(depth)이며, **평탄 접기(flat-foldable)가 성립하려면 반드시 depth == height**, 즉 이 두 값은 애초에 같은 값으로 취급된다. 기본값은 `width=40mm, height=30mm`.

## 작동 방식

### 2D 평면 도안

`hw = width / 2`, `d = height`(= depth)로 두고 좌표를 계산한다.

- **자르기 선(CUT)은 상자의 좌우 세로변만**이다: `(cx-hw, cy-d)~(cx-hw, cy+d)`와 `(cx+hw, cy-d)~(cx+hw, cy+d)`. 위/아래 가로 변(`cy-d`, `cy+d`)은 반드시 안 잘려 있어야 한다 — 그게 상자 패널이 주변 카드에 붙어 있는 지점이기 때문이다. 거기까지 자르면 패널이 완전히 떨어져 나가 팝업을 세울 게 없어진다.
- **척추 산접기(MOUNTAIN_FOLD)**: `(cx-hw, cy)~(cx+hw, cy)` — 상자가 시청자 쪽으로 튀어나오므로 산접기.
- **위/아래 골접기(VALLEY_FOLD)**: `(cx-hw, cy-d)~(cx+hw, cy-d)`와 `(cx-hw, cy+d)~(cx+hw, cy+d)` — 상자가 평평한 카드 면에 실제로 붙는 자리.
- **풀칠 탭(GLUE_TAB)**: 좌우 변에 사다리꼴 두 개.

### 3D 조립 포즈 (열림 각도에 따른 동적 포즈)

box-popup은 v-fold와 완전히 동일한 위상(topology)이다: 척추에서 depth `d`(=box의 `height` 파라미터)만큼 떨어진 두 부착점 + 척추에 있는 공유 크리스 하나. 차이는 뾰족한 삼각형 대신 폭이 있는 직사각형이라는 점뿐이다.

- v-fold와 동일한 대칭 굽힘 각 `a = α/2`를 쓰되, 단일 레벨용 공식 `h = d·sin a`로 `h`를 구하고 `γ = arcsin(h/d)`를 계산한다.
- v-fold와 달리 이론상으로는 **복합 회전이 필요 없다**: 직사각형의 부착 모서리가 이미 척추와 평행(=페이지 자신의 회전 축과 평행)하기 때문에, "플랩의 바깥(부착) 모서리를 피벗으로 삼는 단순한 두 번째 회전"으로 충분해 보인다.
- 하지만 실제 구현은 v-fold 팔과 **동일한 복합 회전 조합**을 그대로 재사용하고 있다 — v-fold의 "검증된" 회전 조합을 그대로 가져다 쓴 것이다(아래 "이전 작업" 참고: 초기엔 단순 회전만으로 시도했다가 실패했다). 즉 서술상의 이론(단순 회전)과 실제 구현(복합 회전 재사용)이 다르므로, 코드를 수정할 때는 **주석이나 서술보다 실제 계산식을 신뢰**해야 한다.
- `a=0`에서 플랩은 이미 edge-on인 자기 페이지와 동일 평면에 있어 숨고, `a=90`(α=180)에서 플랩이 90도 회전해 자유(척추 쪽) 모서리가 시청자를 정면으로 향하며 높이 `d`에 도달한다. 좌우 폭(`width`)이 같고 같은 중심선에 정렬돼 있어 두 플랩의 자유 모서리가 항상 일치해 하나의 이어진 평평한 상자 면을 이룬다.

## 이 프로젝트에서의 활용

3D 조립 포즈 프리뷰가 지원되는 3종 중 하나다. 이 3종은 (v-fold, box-popup, parallel-fold) 순서의 조건 분기 마지막 자리(else 분기)로 box-popup이 처리되는 구조였다 — 새로운 3D 지원 메커니즘을 추가할 때 이 catch-all을 명시적 조건으로 바꿔야, 실수로 box-popup 코드 경로가 다른 메커니즘에 잘못 적용되는 것을 막을 수 있다(3종만 있을 땐 문제 없지만 네 번째 3D 메커니즘 추가 시 위험). 인쇄용 조립 설명서는 v-fold와 마찬가지로 전용 문구 없이 범용 3단계 설명을 사용한다.

## 이전 작업에서 배운 교훈

과거 한 커밋이 **실제 자르기/접기 도안의 버그**를 고쳤다: 원래 코드는 상자의 위/아래 가로 변(`cy-d`, `cy+d`)을 **자르기 선과 골접기 선 둘 다로 동시에** 그리고 있었다(동일 좌표에 CUT과 VALLEY_FOLD 두 스타일을 겹쳐 그림). 조립 설명서가 시키는 대로 모든 실선을 자르면, 상자 패널이 카드에서 완전히 떨어져 나가 팝업을 지탱할 게 아무것도 남지 않는 치명적 버그였다.

수정은 단순히 위/아래 가로 변의 **중복된 CUT 등록을 제거**하고 VALLEY_FOLD만 남긴 것이다. 세로 변(좌우)만 CUT으로 남아 상자 옆면의 실루엣을 만들고, 위/아래는 골접기로 남아 패널이 카드에 붙은 채 접힌다.

같은 시기에 3D 프리뷰도 함께 추가됐는데, v-fold 3D 코드(부모-자식 nesting + 복합 회전 조합)를 거의 그대로 재사용했다. 처음엔 box-popup이 직사각형이라 단순 회전으로 충분할 거라 가정했지만, 실제로는 edge-on(보이지 않는 각도)으로 렌더링되는 실패를 겪었고, 결국 v-fold의 검증된 복합 회전 방식을 그대로 가져와야 실제로 시청자를 향하는 면이 나왔다. α = 0/45/90/135/180에서 시각 검증을 마쳤다.

## 앞으로 작업 시 주의사항

- **동일 좌표에 CUT과 FOLD를 동시에 그리지 않았는지 항상 확인한다.** 이 버그의 근본 패턴(같은 선을 자르기+접기 이중으로 등록)은 parallel-fold에서도 똑같이 반복됐다 — 이 계열 메커니즘에서 반복적으로 나타나는 버그 클래스이므로, 새 메커니즘을 만들거나 기존 것을 수정할 때 **모든 좌표를 cuts/mountainFolds/valleyFolds 배열 간에 교차 검사**하는 습관이 필요하다(동일 좌표 쌍이 두 배열에 동시에 있으면 버그).
- **"완성 시 조립 설명서대로 실선을 다 오리면 패널이 카드에서 완전히 분리되지 않는지" 실물 기준으로 재현해 본다.** 이 버그는 시각적으로는 도안이 그럴듯해 보여도 실제로 오리고 접어봐야 드러난다.
- **`width`와 `height`(=depth)를 독립적으로 크게 벌리는 기능을 추가하기 전에**, "depth MUST equal height for flat-foldability" 제약이 실제로 강제되고 있는지 확인한다 — 원래는 `height` 하나로 `depth`를 대신하고 있어 이 제약이 코드 구조상 자동으로 지켜지지만, 파라미터를 분리하는 리팩터를 하게 되면 이 제약을 명시적으로 검증(clamp 등)해야 평탄 접기가 깨지지 않는다.
- **3D 프리뷰 코드를 고칠 때 서술과 실제 구현이 어긋나 있을 수 있음에 주의**: 서술은 "단순 회전"이라고 말할 수 있지만 실제 코드는 v-fold와 동일한 복합 회전을 쓴다. 코드를 읽을 때 주석만 보고 판단하지 말고 실제 계산식을 반드시 확인한다.
- **`else` 분기로 처리되는 box-popup 3D 코드** — 새로운 3D 지원 메커니즘을 추가할 때 이 catch-all `else`가 의도치 않게 새 메커니즘에 적용되지 않도록, 명시적 조건 분기로 바꾸는 편이 안전하다.
